## 一、自我介绍 + 项目是什么（0:00 — 1:05）

**画面：** 主播面对镜头。

> 大家好，我是 Silas。
>
> 今天给大家带来一个高并发抽奖项目——就是大家平时在淘宝、拼多多上见到的那种，秒杀、抽奖、抢优惠券，同一时间大量用户进来抢同一批库存。
>
> 处理这种场景，业界有两种主流方案。一种是把库存放 Redis，用 Lua 脚本原子扣减；另一种是用 MySQL 行锁扣减，外面套一层 Redis 做旁路缓存。
>
> 网上讨论这两种方案的文章很多，但大多停在理论。到底差多少、什么压力下开始扛不住、扛不住了会怎样——这些你得看到数据才能判断。
>
> 所以我就用 Go 写了一个系统。同一个抽奖业务，里面同时实现了这两套方案，页面上点一下就能切换。然后用同一把压测工具、同一批参数去打，吞吐量、延迟、数据库压力全在一块实时面板上，SSE 一秒一刷。
>
> 代码开源。你想自己学、做毕设、做教学演示，都可以直接用。
>
> 今天这集，我们把它从头跑到尾。

## 三、系统架构（1:45 — 2:35）

**画面：** 架构链路图，逐段点亮。

```
用户请求
  → Gin 路由（/lucky 或 /lucky/cacheaside）
    → 守门员（预扣：令牌桶限流 / 旁路：熔断器）
      → 库存扣减（★ 两种方案唯一的差异点）
        → 拿到资格 → 付款 → MySQL 订单
        → 没付款/超时 → RocketMQ 延时消息 → 释放库存
```

> 请求进来，先到 Gin 路由。选了哪个模式走哪条路径——`/lucky` 是预扣，`/lucky/cacheaside` 是旁路。
>
> 然后是一个守门员。预扣模式用令牌桶限流，控制打到 Redis 的速率。旁路模式用熔断器，MySQL 扛不住了就主动拒绝，避免雪崩。
>
> 再往下是库存扣减，这是两种方案唯一不同的地方。预扣在这里打 Redis Lua 脚本，旁路打 MySQL 行锁。
>
> 拿到资格之后，付款落订单。超时没付，RocketMQ 发延时消息把库存还回去。
>
> 后面所有压测数据，回答的都是同一件事：库存扣减这一步的两种写法，各自有什么后果。

## 四、两种方案怎么工作的（2:35 — 3:50）

**画面：** 分屏示意图，左边预扣数据流，右边旁路数据流。

### 5.1 Redis 预扣库存

> 先说预扣库存。核心思路：库存权威数据放 Redis，扣减不走数据库。
>
> 入口先过一个令牌桶限流器。每秒固定放行一定数量的请求，超出的直接拒绝。这是预扣模式的守门员——控制打到 Redis 的并发量，不让 Redis 也被冲垮。
>
> 限流通过之后，一个 Lua 脚本里做了四件事——检查你有没有重复参与、检查库存、扣库存、写临时资格。
>
> 四件事一次 Redis 调用，原子执行。Redis 单线程处理命令，Lua 脚本中间不会被其他请求插入。
>
> 如果不原子——你先查库存还有，但扣之前别人也查到还有，两个人都去扣，库存就负了。
>
> Lua 只返回三种结果：OK、DUPLICATE、SOLD_OUT。
>
> 在进 Lua 之前，会先查一次 MySQL——看用户是否已有完成订单，这是幂等性校验，一次 SELECT，不参与扣库存。
>
> 用户付款后异步写订单到 MySQL。超时没付，RocketMQ 延时消息把库存还回 Redis。
>
> 所以预扣模式的链路是：Redis Lua 扛库存扣减，MySQL 做幂等校验和持久化，RocketMQ 兜底补偿。

### 5.2 旁路缓存 Cache-Aside

> 再看旁路缓存。标准做法：读的时候先读 Redis，没有再查 MySQL 回填；写的时候更新 MySQL 然后删缓存。
>
> 库存扣减用 MySQL 行锁，`UPDATE ... WHERE cache_stock > 0`，库存到零自然扣不下去，不会超卖。
>
> 问题在哪？在场景。
>
> 旁路缓存是为读多写少设计的——商品详情页，读几千次才写一次，缓存命中率很高。
>
> 但扣库存是每次请求都写。1000 QPS 就是每秒删 1000 次缓存。上一个请求刚删，下一个来读到空，去 MySQL 查完回填，扣完又删。
>
> 缓存一直在被删除，命中率上不去。请求绕过 Redis 直接到 MySQL，行锁排队，连接池打满，响应时间拉长。

## 五、压测：旁路缓存阶梯加压（3:50 — 5:20）

**画面：** 终端敲 wrk2 命令，回车，切回浏览器指标面板。

> 好，讲完旁路缓存的工作原理，我们现在不要一上来就打 1000 QPS。
>
> 因为 1000 QPS 对旁路缓存太猛了，画面会一瞬间红灯，观众还没看清压力怎么传导，系统已经开始熔断了。
>
> 所以这里我用阶梯压测：先 150 QPS 看健康状态，再 300、600 一步步加压，最后再用 1000 QPS 把它打到极限。
>
> 压测工具用的是 wrk2。它的好处是可以按固定 QPS 持续打流量——不是服务能跑多少算多少，而是你指定多少它就打多少。这样更适合观察系统在固定压力下是怎么退化的。
>
> 我先复制 150 QPS 这条指令，到终端运行。

**画面：** 命令高亮。

```
docker compose --profile loadtest run --rm \
  -e TARGET_URL=http://app:5678/lucky/cacheaside \
  -e RATE=150 -e DURATION=20s -e CONNECTIONS=64 wrk2
```

> 压测开始之后，回到页面。注意，我们现在压的是旁路缓存接口，所以别看上面那个预扣库存的面板，那是另一套方案的。我们看下面这个「实时 MySQL 压力」面板。

**画面：** 聚焦旁路缓存指标面板。

> 低压阶段先看三件事：QPS 有没有进来、缓存命中率有没有波动、MySQL 连接池有没有明显压力。
>
> 这个阶段一般不会立刻崩，它的作用不是证明系统扛不住，而是给观众一个基线：旁路缓存在压力不大的时候是正常工作的。

**画面：** 指向 QPS 和 TotalRequests。

> 接着我把压力提到 300 QPS。

**画面：** 复制第二条命令。

```
docker compose --profile loadtest run --rm \
  -e TARGET_URL=http://app:5678/lucky/cacheaside \
  -e RATE=300 -e DURATION=25s -e CONNECTIONS=96 wrk2
```

> 这时候开始观察缓存命中率和库存回源查询。
>
> 旁路缓存的问题不是“完全不能用”，而是在写库存越来越频繁的时候，缓存刚回填就被下一次扣库存删除。命中率开始掉，回源查询开始涨，说明 Redis 挡流量的能力正在变弱。

**画面：** 指向缓存命中率、库存回源查询。

> 然后继续加到 600 QPS。

**画面：** 复制第三条命令。

```
docker compose --profile loadtest run --rm \
  -e TARGET_URL=http://app:5678/lucky/cacheaside \
  -e RATE=600 -e DURATION=25s -e CONNECTIONS=160 wrk2
```

> 这一档开始重点看 MySQL：平均响应、MySQL P95、MySQL P99，还有连接池占用率。
>
> 如果连接池占用开始接近 100%，说明请求不是在业务代码里慢，而是在等数据库连接、等 MySQL 行锁。这个时候入口 P95、P99 会开始明显抖动。

**画面：** 指向 DB 相关指标。

> 这里的抖动不是 bug。因为有的请求真的进了 MySQL，要排队；有的请求被熔断器快速拒绝，反而很快返回。所以入口响应时间会一闪一闪。
>
> 录视频的时候不要追着这个数字念，重点讲趋势：连接池打满、DB P95/P99 变高、熔断拒绝开始增加。

**画面：** 聚焦连接池、DB P95/P99、熔断拒绝。

> 最后我们再上 1000 QPS，看它的极限状态。

**画面：** 复制第四条命令。

```
docker compose --profile loadtest run --rm \
  -e TARGET_URL=http://app:5678/lucky/cacheaside \
  -e RATE=1000 -e DURATION=30s -e CONNECTIONS=256 wrk2
```

> 到这一档，画面通常会很快变红。这个时候就不要再解释每一个数字了，直接讲链路结论——
>
> 每来一个请求，扣一次库存，写一次 MySQL，删一次 Redis 缓存。下一批请求进来，缓存已经没了，只能回源 MySQL。
>
> 于是库存回源查询上升，扣减写入持续打 MySQL，连接池被占满，DB P95、P99 走高，最后熔断器介入。

**画面：** 聚焦连接池占用和 DB 延迟，数字攀升。

> 熔断降级拒绝数开始上涨。注意，这些不是业务成功，是系统在主动快速拒绝——fail-fast，保护数据库不被彻底打挂。

**画面：** 聚焦熔断降级拒绝数。

**画面：** 回到面板全景。

> 所以这次压测看的不是单纯「有多少请求成功了」。看的是旁路缓存这种强一致方案，在写密集场景下，压力是怎样一步步传导的——
>
> 缓存失效，请求打到 MySQL，行锁排队，连接池打满，延迟飙升，最后熔断保护。
>
> 这个过程，就是旁路缓存不适合秒杀热点写库存的核心证据。

## 六、压测：预扣库存 1000 QPS（5:20 — 6:10）

**画面：** 切换到预扣模式，wrk2 重新启动，同样 1000 QPS。

> 现在切到预扣库存，同样 1000 QPS。

### 7.1 QPS 和延迟

**画面：** 聚焦 QPS 和 P95 延迟。

> QPS 稳定 1000。P95 延迟几十毫秒。
>
> 能稳定吃满 1000，一方面是 Redis Lua 处理得快，另一方面是令牌桶限流器在前面把流量削平了——每秒只放固定数量的请求进来，不会让瞬时尖峰把 Redis 也打爆。
>
> P95 的意思是 100 个请求里 95 个低于这个值，比平均值更能反映大多数人的实际体验。

### 7.2 数据库压力

**画面：** 聚焦数据库响应时间和连接池占用率。

> 数据库响应时间基本没变，连接池占用不到 5%。
>
> MySQL 不是完全闲着——它在做两件事：查用户是否已有完成订单，这是幂等性校验；抽中后查奖品详情。
>
> 两次都是 SELECT，不加锁。扣库存这个最重的操作在 Redis Lua 里完成了。

### 7.3 一致性代价

**画面：** 「最终一致 vs 强一致」对照。

> 预扣有代价。Redis 先扣库存，订单异步写 MySQL，中间有时间差。如果系统在这个窗口内挂了，Redis 和 MySQL 的数据可能对不上。
>
> 兜底靠两样——RocketMQ 延时消息超时回补库存；服务重启时从 MySQL 初始库存减已支付订单数恢复 Redis 库存。
>
> 所以这是两种取舍：预扣用一致性换性能，旁路用性能换一致性。秒杀场景一般选性能，金融场景必须选强一致。取决于你的业务。

## 七、参数对比（6:10 — 6:50）

**画面：** 对比表逐行弹出。

| 指标 | Cache-Aside（1000 QPS） | 预扣库存（1000 QPS） | 差距原因 |
|------|------------------------|---------------------|---------|
| 实际吞吐 | 远低于 1000 | 稳定 1000 | MySQL 行锁瓶颈 vs Redis 内存 |
| P95 延迟 | 数百毫秒 | < 50 ms | 等行锁 + 等连接池 vs 单次 Redis Lua |
| 数据库 RT | 数百毫秒 | 基本不变 | 全部 UPDATE 打 DB vs DB 只做 SELECT |
| 连接池占用 | 100% | < 5% | 同上 |
| 缓存命中率 | ≈ 0% | 不适用 | 写删缓存持续失效 vs Redis 是权威源 |
| 熔断器 | 红灯 | 绿灯 | DB 压力触发 vs 不触发 |
| 超卖 | 无（行锁） | 无（Lua 原子） | 两个方案各自保证了 |
| 一致性 | 强一致 | 最终一致（MQ 补偿） | 同步写 DB vs 异步落库 |

> 吞吐和延迟差距的根源在架构——旁路每次扣库存穿透 MySQL，行锁和连接池把并发串行化。预扣把扣库存放在 Redis 内存完成，绕过了这两个瓶颈。
