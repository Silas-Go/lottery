## 四、系统架构：一条请求怎么走的（2:05 — 3:00）

**画面：** 架构链路图（动画或手绘），从「用户请求」开始，逐段点亮每一个节点。口播跟画面同步。

> 在看压测数据之前，得先把这套系统的架构搞清楚。不然后面那些数字为什么这样跳，你没办法真正理解。
>
> 一条抽奖请求进来，走的路径是这样的——

**画面：** 逐段高亮链路。

```
用户请求
  → Gin 路由（/lucky 或 /lucky/cacheaside，根据你选的模式）
    → 守门员（预扣模式走令牌桶限流，旁路模式走熔断器）
      → 库存扣减（★ 整个系统最关键的一步，也是两种方案唯一不一样的地方）
        → 拿到资格 → 付款 → MySQL 落订单
        → 没付款/超时 → RocketMQ 延时消息 → 释放库存
```

> 用户请求进来，先到 Gin 路由。你选了什么模式，它就走到对应的路径——`/lucky` 是预扣库存，`/lucky/cacheaside` 是旁路缓存。
>
> 然后是一个「守门员」。预扣模式这边是一个令牌桶限流器，控制每秒打多少请求到 Redis。旁路模式这边是一个熔断器，一旦 MySQL 扛不住了，它主动掐断，快速失败，不让请求堆在那雪崩。
>
> 再往下，就是整个系统的心脏——**库存扣减**。这是两种方案唯一不同的地方，也是最关键的地方。预扣模式在这一步打 Redis Lua 脚本，旁路模式打 MySQL 行锁。这一步的差异，会传导到后面每一个节点。
>
> 拿到资格之后，付款、落订单。MySQL 在这条链路里只负责写最终结果——订单表。
>
> 如果你超时没付，RocketMQ 发一条延时消息，把库存还回去。这是补偿链。
>
> 整条链路记住一个重点：**两种方案唯一的变量就是「库存扣减」这一步。** 后面所有的压测对比，都是在回答同一个问题——这一步的两种写法，会导致什么样的连锁反应。

## 五、两种方案，理论先行（3:00 — 4:20）

**画面：** 分屏示意图。左边画预扣库存的数据流，右边画旁路缓存的数据流。讲一个亮一个。

### 5.1 Redis 预扣库存

> 先说第一种，Redis 预扣库存。
>
> 它的核心思路一句话：**数据库根本不参与抢库存。**
>
> 库存的权威数据直接放在 Redis 里。用户进来，直接走 Redis。但是——不是简单地读一下库存、再写一下那么粗糙。
>
> 我把四件事，塞进了一个 Lua 脚本里——
>
> 第一，检查你有没有重复参与。第二，检查库存还有没有。第三，扣掉库存。第四，写入你的临时资格。
>
> 四个动作，一次 Redis 调用，原子执行，中间没有任何间隙。
>
> 为什么非要原子？你想——如果分开做，你先查库存，发现还剩一个。在你扣之前，另一个请求也查到了还剩一个。两个人都觉得自己抢到了，都去扣。库存变成负的。这就是高并发下最经典的竞态条件。
>
> 放在同一个 Lua 脚本里，Redis 单线程执行，这四个动作中间没有任何东西能插进来。「查的时候有、扣的时候没了」这种事，直接被消灭了。
>
> Lua 只返回三种结果：OK、DUPLICATE、SOLD_OUT。成功、重复参与、库存卖光。干净利落。
>
> 注意，在进 Lua 之前，系统会先查一次 MySQL——看你有没有已经完成支付的订单，这是幂等性校验。一次轻量 `SELECT`，不参与扣库存。真正的高并发压力，全在 Redis Lua 这一步。
>
> 等你真付了钱，才异步写一条订单到 MySQL。你超时没付，RocketMQ 发延时消息把库存还回 Redis。
>
> 所以这套方案的精髓就是：**Redis Lua 扛住所有高并发的库存扣减，MySQL 做幂等性校验和最终持久化，RocketMQ 兜底补偿。** 扣库存这个最重的操作，碰都不碰数据库。

### 5.2 旁路缓存 Cache-Aside

> 再看第二种，Cache-Aside，旁路缓存。教科书级别的缓存模式，写后端的人应该都听过。
>
> 规矩很简单：**读的时候，先读 Redis，没有就查 MySQL，然后回填缓存。写的时候，直接更新 MySQL，然后删除缓存。**
>
> 听起来很标准，而且它有一个很多人不知道的好处——**它不会超卖。** 因为我用的是 MySQL 行锁，`UPDATE` 的时候 `WHERE` 条件里写了 `cache_stock > 0`。库存到零就扣不下去了。数据库这一关守得死死的。
>
> 但是——它有一个**场景不适配**的问题，而且是致命的。
>
> 旁路缓存是为「读多写少」设计的。商品详情页、用户信息——读几千次才写一次，缓存热乎乎的，命中率 90% 以上。
>
> 可「扣库存」是什么？**每一次请求都是一次写。** 1000 QPS 就是每秒 1000 次写。每一次写，删一次缓存。
>
> 上一个请求刚删了缓存，下一个请求来了——读，空的。然后它去查 MySQL，回填缓存，再扣库存——又写，又删。
>
> 缓存永远暖不起来。这个现象不是偶发的缓存击穿，是**机制性的——写删缓存的逻辑本身，在写密集场景下就是自毁的。**
>
> 结果就是：缓存形同虚设，所有请求绕过 Redis，直接砸到 MySQL 上。行锁排队、连接池打满、响应时间飙升。
>
> 好，理论讲完了。下面直接上 1000 QPS，看数据。
>
> 先说明一点——今天这轮压测，**只压秒杀入口，不模拟完整的支付闭环。**
>
> 为什么？因为在秒杀系统里，真正把系统打爆的，就是**入口这一瞬间的流量和库存扣减**。1000 个人同时抢，库存扣减扛不住，后面的支付链路再好都没用。
>
> 支付链路——只有抢到资格的那一小部分用户才会走到。后续可以单独做场景压测，把支付率、放弃率、超时比例也编排进去，那是另一种维度的测试了。今天先聚焦核心：**库存扣减这一步，两种方案到底差多大。**

## 六、压测实战 · Cache-Aside 翻车现场（4:20 — 6:00）

**画面：** 切换到 Cache-Aside 模式。终端启动 wrk2，1000 QPS。右侧指标面板数字开始跳。依次聚焦每个指标——先解释它是什么，再看它怎么跳，最后归因。

### 6.1 缓存命中率——接近零

**画面：** 聚焦「缓存命中率」。

> 第一个指标：**缓存命中率。**
>
> 它说的是——请求来的时候，有多少能在 Redis 里直接找到数据，不用去查 MySQL。正常一个成熟的缓存系统，命中率在 80%、90% 以上。越高越好，因为 Redis 扛住了大部分读。
>
> 现在你看——**几乎为零。**
>
> 为什么？就是刚才说的：每一次扣库存都是写，每一次写都删缓存。缓存刚暖就删，刚暖就删，永远起不来。
>
> 这个零不是偶然的毛刺，是**稳态**——在这个场景下旁路缓存的缓存命中率，稳态就是零。这是写密集对 Cache-Aside 的降维打击。

### 6.2 数据库响应时间 + 连接池——飙升

**画面：** 聚焦「数据库响应时间」和「连接池占用率」，两条线快速攀升。

> 第二个指标：**数据库响应时间**，就是 MySQL 处理一次查询要花多久。正常几毫秒到十几毫秒。你看现在——几十毫秒、上百毫秒，还在涨。
>
> 旁边这个，**连接池占用率**，直接拉满到 100%。
>
> 为什么会这样？因为缓存是空的，请求全砸到 MySQL 了。MySQL 在处理什么？每一个扣库存是一个 `UPDATE`，`UPDATE` 要拿行锁——同一行数据，同一时刻只能有一个人改。1000 个请求排队等同一把锁。
>
> 一个人拿到锁，改完，释放。后面的抢到锁，改完，释放。再后面再抢。这个叫**行锁竞争把并发串行化了**——本来应该并行的 1000 个请求，被锁逼着排队。
>
> 连接池也一样——一个连接处理一个请求，连接用完了，新来的请求连 MySQL 的门都进不去，只能在外面排队干等。

### 6.3 熔断器——绿 → 黄 → 红

**画面：** 聚焦「熔断器」。

> 第三个指标：**熔断器。**
>
> 这是系统最后一道防线。它不是等「请求报错了」才反应——那是事后诸葛亮。这个熔断器看的是**数据库的实时压力**。
>
> 两个触发条件：数据库响应时间超过 100 毫秒，或者连接池占用率超过 80%。连续触发五次，直接跳红——**熔断。**
>
> 红灯的意思是：主动拒绝新请求，直接返回失败。宁可让用户看到「系统繁忙」，也绝不让请求在数据库那里堆成雪崩。
>
> 三秒冷却后，放几个试探请求进去——这就是 Half-Open，半开状态。压力降下来了就回绿灯，没降下来继续红。
>
> 这不是「出错了就断开」，是一个**压力感知的自愈开关**。它会自己恢复。

### 6.4 小总结

**画面：** 回到面板全景，字幕打出 Cache-Aside 1000 QPS 的结论。

> 好，Cache-Aside 1000 QPS 的结果——
>
> 缓存命中率，零。数据库响应时间，几百毫秒。连接池，打满。熔断器，跳红。
>
> 但是它**没有超卖**——MySQL 行锁把库存守住了。它的代价不是数据错，是**慢**——慢到系统不可用。
>
> 这不是 Cache-Aside 这个模式本身有问题，是「扣库存」这种写密集的场景，天生就不适合用它。

## 七、压测实战 · 预扣库存 1000 QPS 稳如狗（6:00 — 7:00）

**画面：** 切换到预扣库存模式。终端重新启动 wrk2，同样 1000 QPS。面板刷新。可做分屏——左边刚才 Cache-Aside 的翻车回放，右边现在预扣全绿。

> 好，现在切到预扣库存。**同样 1000 QPS，同样的业务，同一个 wrk2。**

### 7.1 QPS 和延迟

**画面：** 聚焦 QPS 和 P95 延迟。

> 看 QPS——稳稳吃满 1000。再看 P95 延迟——几十毫秒。
>
> 顺便说一下，P95 是什么意思？它说的是 100 个请求里，有 95 个的响应时间低于这个值。这个数比平均值更有意义——因为平均值会被少数极慢的请求拉高，P95 反映的是「绝大多数人感受到多快」。
>
> 预扣模式 P95 几十毫秒，说明绝大多数请求进来，Redis Lua 脚本瞬间执行完就返回了。没有排队，没有等锁，没有穿透到数据库。

### 7.2 数据库压力——只有轻量查询

**画面：** 聚焦数据库响应时间和连接池占用率，全程低位。

> 再看数据库这边——响应时间基本没动，连接池占用率不到 5%。
>
> 但注意，MySQL 不是完全没动。它确实在被查——每来一个请求，系统会查两样东西：
>
> 第一，**查你有没有已经完成支付的订单**——这是幂等性校验，防止付过钱的人再来抽。
>
> 第二，抽中之后，**查一下这个奖品的详细信息**——叫什么、多少钱、图片链接。
>
> 两次都是 `SELECT`。轻量查询，不加锁，毫秒级返回。这和 Cache-Aside 那边每次都要拿行锁做 `UPDATE` 扣库存，完全不是一个量级。
>
> 这就是预扣模式最核心的优势：**扣库存这个最重的操作，全程在 Redis 内存里完成。** 1000 QPS 的写压力全由 Redis 的 Lua 原子脚本扛住，MySQL 只做两件事——轻量的幂等性校验、和最终的订单持久化。热路径和冷路径彻底分离，互不拖累。

### 7.3 代价是什么——一致性

**画面：** 字幕打出「最终一致 vs 强一致」，配 RocketMQ 延时消息动画。

> 但预扣模式有没有代价？当然有。**一致性。**
>
> 它是最终一致，不是强一致。Redis 先扣了库存，订单异步写到 MySQL。中间有几毫秒到几秒的时间差。如果系统刚好在这个窗口里挂了，Redis 里的扣减记录和 MySQL 里的订单之间，可能对不上。
>
> 怎么兜底？两样东西——**RocketMQ 延时消息**，超时没付款自动把库存还回去；**服务重启时的库存恢复逻辑**，从 MySQL 初始库存减去已支付订单数，把 Redis 库存回正。
>
> 所以，预扣库存是用**一致性换性能**。旁路缓存是用**性能换一致性**。
>
> 绝大部分秒杀场景选性能——库存本来就要被抢光的，短暂不一致能接受，补偿链兜得住就行。但如果你的场景是银行转账——那每一笔都必须是强一致，旁路缓存反而是对的。关键是你知道自己在换什么。

## 八、双模式参数全景对比（7:00 — 7:45）

**画面：** 一张对比表，数据一行一行弹出。每弹一行，简单解释差距原因。

| 指标 | Cache-Aside（1000 QPS） | 预扣库存（1000 QPS） | 为什么差这么多 |
|------|------------------------|---------------------|--------------|
| 实际吞吐 | 远低于 1000 | 稳稳 1000 | 瓶颈在 MySQL 行锁 vs 瓶颈在 Redis 内存 |
| P95 延迟 | 几百毫秒 | < 50 ms | 等行锁 + 等连接池 vs 单次 Redis Lua |
| 数据库 RT | 几百毫秒 | 基本不动 | 全部请求打 DB vs DB 只在支付时异步写 |
| 连接池占用 | 100% | < 5% | 同上 |
| 缓存命中率 | ≈ 0% | 不适用 | 写删缓存 vs Redis 就是权威源，不走缓存 |
| 熔断器 | 红灯 | 全程绿灯 | DB 压力触发 vs 不打 DB，没东西触发 |
| 超卖 | 无（行锁防住） | 无（Lua 原子防住） | 两个方案都对了 |
| 一致性 | 强一致 | 最终一致（MQ 补偿） | 同步写 DB vs 异步落库 |

> 几个点单独说一下——
>
> **吞吐和延迟的差距，根子在架构。** Cache-Aside 每一步扣库存都穿透到 MySQL。MySQL 的行锁和连接池是两个串行化的瓶颈——并发越高排队越严重。预扣模式把这个瓶颈直接绕过去了——Redis 单线程但纯内存，Lua 原子执行没有锁竞争，吞吐只受 Redis 自身能力限制。
>
> **一致性的选择。** Cache-Aside「每一笔立刻写对」，代价是慢。预扣「先快速扣了再慢慢对齐」，代价是中间有一个不一致的窗口。你选什么，取决于你的业务能不能接受那几毫秒的最终一致。
>
> **熔断器的意义。** Cache-Aside 需要熔断器，因为它知道自己会压到数据库。预扣不需要——不是预扣更聪明，是它根本不走那条路。最好的保护不是「压力大了再掐断」，而是「压根不把压力传过去」。
