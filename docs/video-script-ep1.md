> 我是 Silas。
>
> 这是我用 Go 写的一个高并发抽奖系统。不过它的重点其实不在抽奖本身，而是真实系统里最核心的那一块——**秒杀、支付这类高并发链路的架构**。

## 带他们跑起来（0:20 — 0:55）

**画面：** 切到终端。依次敲 `.\scripts\start-infra.ps1`（Docker 拉起 MySQL / Redis / RocketMQ）、`.\scripts\run-local-app.ps1`（本机跑 Go app）。日志刷完、监听 `localhost:5678` 那一行给个高亮。

> 项目怎么跑，一句话：**依赖放在 Docker 里，Go 进程跑在本机**。
>
> 一条命令，把 Docker 拉起来——MySQL、Redis、RocketMQ 都准备好；再一条命令，本机就把 Go app 跑起来了。
>
> 这三个各管一段：MySQL 存最终的订单，Redis 扛高并发的库存，RocketMQ 管超时、把没付款的库存还回去。它们分别在哪儿发力，后面你就看到了。
>
> 跑起来之后，浏览器打开 `localhost:5678` 就行。

## 长什么样 · 介绍关键部分（0:55 — 1:30）

**画面：** 切到浏览器。先给抽奖转盘一个全景，然后鼠标依次指向三个关键区域：①左上角「库存模式切换」；②中间抽奖转盘；③右侧实时指标面板。

> 打开就是这个抽奖转盘。我点一下，抽中一个奖品——走的就是一整套流程：抽奖拿到资格、去付款、付了就成单、超时没付，库存就放回去。
>
> 接下来看三个地方，这一集你记住这三个就够了——
>
> **第一，左上角这个模式切换。** 它是整个项目的总开关：预扣库存，还是旁路缓存。同一套业务，点一下就换一套库存实现。
>
> **第二，中间这个转盘。** 每个奖品被抽中的概率，是按它当前还剩多少库存实时算出来的——库存越多，越容易被抽到，这样奖品能尽量发完。
>
> **第三，右边这块实时指标面板。** QPS、P95 延迟、数据库响应时间、连接池占用、熔断器现在亮什么灯、有没有超卖——全部一秒刷新一次，用 SSE 实时推过来。**后面两套方案差在哪，就靠盯着这块面板看。**

## 参数一眼看懂（1:30 — 2:00）

**画面：** 分屏或字幕标注。左边浏览器，右边把三个关键参数用大字打出来（不铺整张表）。

> 项目里能调的参数有一堆，大部分你都不用管，README 里都写着。真正影响这一集演示的，就三个——
>
> **第一个是压测速率**，就是每秒往接口打多少个请求，`RATE`。我会从 500、1000 一路往上加，看谁先扛不住。
>
> **第二个是这条黄线和红线**：数据库响应时间的预警值和熔断值。慢到黄线，面板就报警；超过红线，直接熔断。默认黄线 30 毫秒，红线 100 毫秒。
>
> **第三个是连接池的并发上限**，模拟数据库同时能处理多少请求；调小一点，瓶颈更容易压出来。
>
> 记住这三个就够了：**打多快、什么时候报警、连接池多大**。

## 那么，问你一个问题（2:00 — 2:20）

**画面：** 转盘页面 + 高并发请求涌入动效。字幕大字「会超卖吗？」，随即打个红叉，旁边写「不会 → 会拖垮 DB」。

> 好，东西你也看清楚了。现在，我问你一个问题。
>
> 高并发抢库存，如果用最经典的旁路缓存，也就是 Cache-Aside，你觉得会出什么问题？
>
> 很多人第一反应是：会超卖吧？
>
> 不对。而这，恰恰是我做这个项目想纠正的一个误区——
>
> Cache-Aside 扣库存，**根本不会超卖**。它真正的问题更隐蔽：它会把你的数据库，活活拖垮。
>
> 为了把这件事说清楚，接下来我把两套方案摊开来讲。

## 项目定位（2:20 — 2:40）

**画面：** 回到「两种库存模式对决」那张对比表，`唯一的变量` 那几行高亮。

> 再说一遍这个实验台的思路：**两种模式的业务流程完全一样**——抽奖、给资格、付款成单、超时释放。
>
> **唯一不一样的，就是扣库存这一步**——谁来扛、怎么扣、扛不住了又怎么办。
>
> 我把两种最主流的做法都写了进去，就是想亲眼看看，同一把尺子下，它们到底差多少。

---

## 方案一：Redis 预扣库存（2:40 — 3:30）

**画面：** 代码 `internal/database/admission.go` 的 Lua 脚本片段高亮，「三件事」逐条打字幕。

> 先说第一种，Redis 预扣库存。
>
> 核心就一句话：**数据库根本不参与抢库存**。
>
> 用户进来，直接走 Redis。但不是简单读一下、再写一下那么糙。
>
> 我用一段 Lua 脚本，在一次 Redis 调用里，原子地把三件事一起做完——
>
> 看你有没有重复参与；看库存还有没有；然后扣掉库存、写入你的临时资格。
>
> 三件事，一个原子操作，拆不开。
>
> 「查的时候还有、扣的时候没了」这种经典的竞态，在 Lua 这一层就直接没了。脚本只会返回三种结果：成功、重复、卖光。
>
> 这期间 MySQL 一直不动。等你真付了钱，才落一条订单。要是超时没付，RocketMQ 会发一条延迟消息，悄悄把库存还回 Redis。
>
> 所以这套方案的精髓就是：**高并发的压力全压在 Redis 上，MySQL 只负责写最终结果**。

---

## 方案二：旁路缓存 Cache-Aside（3:30 — 4:25）

**画面：** 代码 `internal/service/lottery_cacheaside.go` + MySQL 行锁 `UPDATE` 语句；随后切熔断器三态示意图（绿/黄/红）配响应时间曲线。

> 再看第二种，Cache-Aside，旁路缓存，教科书级别的缓存模式。
>
> 读的时候，先读 Redis，没有再去查 MySQL、然后回填缓存。写的时候，直接改 MySQL，再把缓存删掉。
>
> 听着挺标准。可一旦放到**写特别多**的扣库存场景，问题就来了——
>
> 每扣一次库存都是一次写，每写一次，就删一次缓存。
>
> 高并发下，缓存几乎永远是冷的。请求一批一批绕过 Redis，直接砸到 MySQL 上。
>
> 这里我得再强调一下那个误区：它**不会超卖**。因为我用的是 MySQL 行锁，`UPDATE` 的时候库存大于零才扣，数据库这一关守得死死的。
>
> 它的代价不是超卖，是**慢**。行锁排队、连接池被打满、响应时间飙升。
>
> 为了让这个瓶颈能被看见，我给这条链路加了两样东西：一个模拟连接池的**并发闸门**，还有一个**压力感知的熔断器**。
>
> 这个熔断器不看错误率，它盯的是 MySQL 的响应时间和连接池占用率。绿灯放行，黄灯预警，红灯就直接熔断、拒掉新请求，保护数据库。
>
> 这才是旁路缓存在高并发下，真正会撞上的那堵墙。

---

## 为什么值得做这个对比（4:25 — 5:05）

**画面：** 实时指标面板录屏（`/api/metrics/stream`），数字跳动；穿插 wrk2 压测命令行 + `Requests/sec` 输出。

> 你可能会说：那不就完了嘛，Redis 预扣肯定更快，还比什么？
>
> 可是——快多少？到哪个 QPS，Cache-Aside 开始扛不住？熔断器是几秒跳的、跳完能不能自己缓过来？Redis 预扣那条异步补偿链，压到极限，会不会出一致性问题？
>
> 这些，光看文章是看不出来的。
>
> 还有一个我特别喜欢的细节：两种模式共用**同一张库存表**，只是读写不同的列，互不干扰。所以这把尺子，是绝对公平的。
>
> 前面你看到的那块实时面板，就用 SSE 一秒刷一次——QPS、P95 延迟、数据库响应、连接池占用、熔断器亮什么灯、有没有超卖，全都实时看得见。
>
> 然后拿 wrk2 固定 QPS 往上压，500、1000、2000，一边压一边看数字跳。
>
> 不是嘴炮，是数据。
