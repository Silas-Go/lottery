# 视频脚本：同一秒杀订单，两种交易写路径

## 本集结论

本集不再做“Redis 预扣 vs Cache-Aside 库存”的伪单变量对比，而是回答：

> 同一个秒杀业务，MySQL 同步交易路径与 Redis 准入、MQ 异步落单路径分别解决什么问题，又付出什么一致性成本？

## 开场

画面：同一个转盘、同一个订单状态面板、两种模式切换。

口播：

> 两套架构首先必须做成同一件事。今天无论选择 MySQL 还是 Redis，订单都只走这一条生命周期：库存获取、待支付、已支付，或者取消。支付和取消是互斥终态，库存只能回补一次。业务语义不统一，所有性能数字都没有比较价值。

## 第一部分：统一状态机

展示：

```text
stock_acquired
-> pending_payment
-> paid
 \-> cancelled
```

口播要点：

- `stock_acquired` 表示库存已经锁定，但正式待支付订单可能还没建立。
- `pending_payment` 才允许支付。
- `paid` 永不回补库存。
- `cancelled` 必须回补且只能回补一次。
- 支付后取消应该进入退款状态，不允许把 `paid` 直接改成 `cancelled`。

演示：打开支付页，Redis 模式先显示 `stock_acquired`，普通 MQ 消费后自动变成 `pending_payment`。

## 第二部分：方案 A——MySQL 同步交易路径

链路：

```text
请求
-> MySQL 条件扣减库存
-> 同一事务创建 pending_payment
-> 返回待支付
```

口播：

> MySQL 是库存权威。条件更新防止库存扣成负数，库存扣减和待支付订单必须处于同一个事务。这样请求成功时，两件事一定同时成立：库存被占用，订单也已经存在。

展示指标：

- DB 写入延迟。
- 连接池占用。
- P95/P99。
- 热点行竞争。

明确 Cache-Aside：

> 页面可能先从 Redis 读取一份候选库存快照，但这份缓存不负责正确性。每次真正准入仍然回到 MySQL 条件更新。写密集场景频繁失效缓存，正好说明 Cache-Aside 更适合商品、活动、订单历史等读场景，而不是库存扣减方案。

## 第三部分：方案 B——Redis 准入与 MQ 异步落单

链路：

```text
请求
-> Redis Lua 原子准入
-> stock_acquired
-> CREATE_ORDER 普通消息
-> MySQL pending_payment
```

口播：

> Redis 一次 Lua 完成防重复、检查库存、扣库存和写入 stock_acquired。请求不等待 MySQL 写订单，而是把订单意图发送到普通消息队列。Consumer 按数据库能承受的速度创建待支付订单，这一步才是 MQ 的异步削峰。

强调两个 Topic：

| Topic | 作用 |
|---|---|
| `CREATE_ORDER` | 普通消息，异步落单、缓冲削峰 |
| `CANCEL_ORDER` | 延迟消息，支付超时检查 |

口播：

> 延迟消息只是 RocketMQ 的附加能力。普通消息的解耦、缓冲、至少一次投递，才是所有主流 MQ 都有的普适价值。

## 第四部分：支付和取消为什么不会同时成功

MySQL 模式：

```text
UPDATE orders
WHERE status = pending_payment
```

Redis 模式：

```text
porder state = pending_payment
-> paid
或
-> cancelled + 回补库存
```

口播：

> 两个请求都以 pending_payment 为前置状态。谁先完成条件状态迁移，谁获得终态；失败方读取结果后幂等结束。因此不会出现订单已经 paid，超时任务又把库存加回去。

演示建议：

1. 同时点击支付并触发取消。
2. 重复发送取消消息。
3. 重复发送创建消息。
4. 展示最终只有一个订单、一次库存变化。

## 第五部分：公平地看性能

不要说：

> Redis 比 MySQL 快多少。

应该说：

> 这是数据库同步交易路径与内存准入、消息队列异步结算路径的架构策略比较。

两套方案同时改变了库存权威和订单写入时机。若要做单变量归因，应拆成：

1. 固定订单写入方式，只比较 MySQL 准入和 Redis 准入。
2. 固定 Redis 准入，只比较同步落单和 MQ 异步落单。

统一观测指标：

- 入口吞吐和 P95/P99。
- `stock_acquired -> pending_payment` 落单延迟。
- MySQL 峰值写 QPS。
- MQ 积压深度。
- 最终支付成功数。
- 取消数和库存回补数。
- Redis/MySQL 状态差异。

## 第六部分：诚实展示最终一致性成本

口播：

> Redis 加 MQ 不是免费性能。它把一个数据库事务拆成了 Redis、MQ、MySQL 三套系统协作。于是我们必须处理重复消息、迟到消息、消息积压、Redis 成功但消息未发出，以及最终账本暂时落后等问题。

当前代码已经覆盖：

- 重复创建幂等。
- 重复取消不重复回补。
- 支付和取消互斥。
- 迟到创建不复活取消订单。
- 业务失败不 Ack，等待 MQ 重试。

仍应明确未完成：

- Redis 准入成功、发送 MQ 前崩溃的可靠事件/outbox。
- 外部支付流水与退款。
- Redis/MySQL 自动对账和告警。
- 多实例全局限流。

## 结尾

口播：

> MySQL 同步方案的价值是权威清晰和事务简单，代价是入口直接承受数据库瓶颈。Redis 加 MQ 的价值是快速准入和削平写峰值，代价是最终一致性和更复杂的故障治理。真正经典的秒杀项目，不是中间件越多越好，而是业务语义统一、状态迁移可证明、每一种复杂度都有明确收益。
